Projects

Business KPI Dashboard (SQL)

SQL + Dashboard

Built a sales-performance KPI dataset in SQL to answer “what changed and why?” across revenue, orders, and customer behavior. The goal was to replace manual spreadsheet summaries with repeatable queries and sanity checks.

Analysis

  • Problem: No consistent source of truth for KPIs; metrics were recalculated manually and disagreed across reports.
  • How it works: Cleaned and joined 4-6 relational tables (orders, order_items, products, customers, dates) and produced a KPI view (daily/weekly) with revenue, MoM growth %, AOV, top categories, and repeat-customer rate.
  • Tools: SQL (CTEs, joins, window functions), SQLite/MySQL, CSV export for dashboard screenshots.
  • Impact (expected): Target: cut weekly KPI refresh from ~30 minutes to ~10 minutes by using saved SQL queries and a repeatable export workflow, plus catch 2-3 data-quality issues per refresh (duplicates, missing dates, invalid totals).

Power BI Executive Summary

Power BI

Designed a Power BI report that turns a raw dataset into an executive-ready view, with drill-through pages and an “insights + actions” summary. The focus is clarity: what changed, what's driving it, and what to do next.

Analysis

  • Problem: Stakeholders could see totals but couldn't quickly explain drivers (region/product/channel) or detect anomalies.
  • How it works: Built a star-schema model, created 8-12 measures (DAX) for KPIs like revenue, margin %, conversion/retention (dataset-dependent), and added drill-through from the exec page to detail pages for root-cause analysis.
  • Tools: Power BI (Power Query, DAX, relationships), Excel/CSV for inputs, optional SQL for preprocessing.
  • Impact (expected): Target: combine 3 common stakeholder views into 1 report (1 executive page + 2 drill-through pages) and reduce “what changed?” analysis time from ~20 minutes to ~5 minutes using consistent DAX measures and drill-through.

Alert Triage Casebook

SOC

A SOC Tier 1-style casebook of alert investigations (true positives + false positives) with structured notes, evidence, and escalation decisions. The goal is to demonstrate a consistent triage workflow and clean handoffs.

Analysis

  • Problem: New analysts often know tools but struggle to document investigations clearly and consistently under time pressure.
  • How it works: Each case includes: alert context, timeline, key evidence (log lines/screenshots), hypothesis, disposition (TP/FP/benign), and an escalation note written for Tier 2/IR.
  • Tools: SIEM-style queries (KQL/Kibana-style depending on lab), Windows Event Logs/Sysmon or network logs (lab), plus a repeatable triage template in Markdown.
  • Impact (target): Build 8 documented alert investigations across 4 categories (e.g., brute force, phishing, malware, suspicious PowerShell), using a one-page triage + escalation template to keep low/medium cases within ~10-15 minutes.

SOC Focus

Tier 1 workflow I practice

  • Monitor + triage alerts: validate signal vs noise and assign severity based on context.
  • Investigate quickly: correlate auth, endpoint, and network logs to build a short timeline.
  • Decide + document: mark TP/FP/benign and write concise notes that another analyst can reuse.
  • Escalate with evidence: include impact, scope, key indicators, and recommended next steps.

Tools, log sources, and artifacts

  • Log sources: Windows Event Logs / Sysmon (lab), basic network logs (DNS/HTTP), and authentication events.
  • Tools: SIEM-style querying (lab), GitHub for documentation, and small scripts (Python) to normalize/summarize logs.
  • Artifacts: triage template, escalation note template, and simple playbooks for common alerts (phishing, brute force, malware).
  • Goal: consistent investigations under time pressure while keeping notes audit-friendly and clear.
